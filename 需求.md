（1）用户管理

**定位与目标**  
用户管理模块是管理员管控系统用户的核心入口，在全平台范围内集中维护账号生命周期与安全策略，避免异常或恶意账号滥用检测与存储资源。

**数据存储**  
用户账号与业务相关属性持久化至 **SQLite** 单文件数据库（与纯 JSON 文件方案相比，便于并发写入、条件查询与后续扩展）。用户主数据采用关系表字段存储；若未来需要存放非结构化扩展信息，可使用 SQLite 的 **TEXT 类型保存 JSON 文本**，并配合内置 **JSON1** 扩展提供的 `json_extract` / `json_valid` 等函数做查询与校验（SQLite 无独立 “JSON 类型”，业界惯例为 TEXT + JSON 函数）。

**功能需求（管理员侧）**  
- **用户列表**：分页或滚动展示全部注册用户，字段至少包括：用户名、真实姓名（若有）、注册时间、账号状态（启用 / 禁用）、累计检测次数（与业务统计关联，可按用户维度汇总）。  
- **账号状态**：对异常账号执行 **启用 / 禁用**；禁用后禁止登录与调用需鉴权的接口。  
- **密码与安全**：支持 **重置登录密码**（由管理员发起或用户自助，具体交互在实现阶段确定）；生产环境密码应 **哈希存储**（如 PBKDF2 / bcrypt），传输层使用 HTTPS。  
- **删除与清理**：支持删除无效或重复账号；删除前宜二次确认，并处理与该用户关联的检测历史策略（保留匿名化记录或级联策略需产品确认）。  
- **审计**：用户管理相关操作（谁、何时、对哪个账号、何种操作）写入 **系统日志**，保证可追溯。

**与普通用户流程的关系**  
前台用户注册、登录仍面向业务用户开放；管理员通过独立后台入口进入用户管理界面（权限模型：管理员角色与普通用户角色分离）。用户管理模块流程图如图 4-8 所示。

（2）病虫害知识库管理

**定位与目标**  
病虫害知识库管理模块用于保障系统内植保类内容的 **准确性、时效性与可检索性**，向普通用户提供可靠的 **茶叶病虫害识别与防治** 知识；管理员在该模块内完成条目的全生命周期维护，避免陈旧、错误信息误导用户。

**数据存储（建议）**  
- **结构化条目**：每条病虫害记录建议落库至 **SQLite**（或等价持久化层），字段与业务一一对应，便于列表筛选、关键词检索与排序；大段文字（如「发病规律」「防治方法」）使用 `TEXT` 存储即可。  
- **参考图片**：图片文件存于服务器约定目录（如 `uploads/knowledge/`），数据库中仅存 **相对路径 / 访问 URL、文件名、可选说明**，避免将二进制直接写入数据库；上传时需校验类型与大小，防止恶意文件。  
- **与现有资料的关系**：当前系统若仍提供 **PDF 图谱/书籍** 在线阅览，可作为「参考资料」与结构化条目 **并存**：PDF 继续按文件管理；结构化知识库侧重单病单虫的条目化查阅与检索。二者在需求上可区分展示入口（「条目库」与「电子资料」）。

**条目数据模型（管理员维护字段）**  
每条记录至少应支持以下信息（实现时可微调字段名）：  
| 字段含义 | 说明 |
|----------|------|
| 病害 / 虫害名称 | 主标题，用于列表与检索 |
| 类型 | 可选：病害、虫害、生理害等，便于分类筛选 |
| 发病症状 | 文字描述，可配合参考图 |
| 发病规律 | 发生时期、环境条件等 |
| 危害特征 | 对产量、品质、树势等的影响 |
| 防治方法 | 农业、物理、化学防治要点及注意事项 |
| 参考图片 | 0～N 张，支持上传、替换、删除 |
| 状态 | **上架 / 下架**：下架后前台不可见，保留数据便于修订后再次上架 |
| 创建 / 更新时间 | 用于排序与审计 |

**功能需求（管理员侧）**  
- **新增**：录入上述字段并保存；图片可多图上传，与条目绑定。  
- **编辑**：修改任意字段；更新防治方案或替换图片后，**更新时间**自动刷新。  
- **删除**：物理删除或逻辑删除（推荐逻辑删除 + 状态「下架」组合，避免误删）；若物理删除，需同步删除孤儿图片文件。  
- **列表与检索**：按名称关键词、类型、状态（上架 / 下架）、时间范围筛选；支持分页。  
- **审计**：新增 / 修改 / 删除 / 上下架等操作宜记入 **系统日志**（与需求（4）衔接），记录操作者与目标条目标识。

**与普通用户流程的关系**  
前台「茶树病虫害知识」类页面面向 **只读**：用户浏览已 **上架** 的条目，可按分类或搜索查看详情与配图；不提供匿名用户直接改库。仅 **管理员账号** 可进入后台知识库管理界面。病虫害知识库管理模块流程图如图 4-10 所示。

（3）系统日志管理

**定位与目标**  
系统日志管理模块用于 **持久化记录** 系统运行期内的关键行为与异常事件，为 **故障排查、安全审计与合规追溯** 提供依据。日志应能回答：谁在何时、对何种资源、做了何种操作、结果是否成功；异常类日志还应包含可定位错误的上下文信息（接口路径、错误摘要等），避免仅依赖控制台输出。

**数据存储（建议）**  
- **结构化日志表**：建议统一落库至 **SQLite**（可与用户、知识库共用同一库文件或分表），每条日志一行，字段固定、可索引，便于按时间、类型、操作者筛选。  
- **扩展信息**：请求参数摘要、业务对象 ID、堆栈片段等可放入 **`TEXT` 字段**（纯文本）或 **`TEXT` + JSON 字符串**（需查询时用 JSON1 函数解析）；单条日志体积不宜过大，敏感字段（如密码、完整请求体）应 **脱敏或不记录**。  
- **保留策略**：可配置 **保留天数或最大条数**，超限时按时间滚动删除或归档；生产环境可另接文件日志 / 集中式日志系统，本需求以应用内查询与导出为主。

**日志类型与内容要点**  
| 类型 | 说明 | 典型字段示例 |
|------|------|----------------|
| **管理员操作** | 后台对用户、知识库等的增删改、启停账号、重置密码等 | 操作者账号、动作编码、目标类型与 ID、结果、时间 |
| **用户行为** | 登录成功/失败、注册、检测请求（可按「每次检测」记一条摘要） | 用户名或匿名标识、动作、资源摘要、时间、是否成功 |
| **系统异常** | 未捕获异常、接口 5xx、关键业务断言失败 | 级别（error/warn）、模块、消息摘要、可选堆栈片段、时间 |

**功能需求（管理员侧）**  
- **日志查询**：按 **时间范围**、**日志类型**、**操作者（用户名）**、**关键词**（匹配消息摘要）组合筛选；列表 **分页** 展示，避免一次加载全表。  
- **详情查看**：单条日志可展开查看完整描述与扩展字段（JSON 可读化展示）。  
- **导出**：支持将当前筛选结果导出为 **CSV 或 Excel**（与论文/答辩演示兼容），导出文件命名包含日期范围。  
- **权限**：仅 **管理员角色** 可访问日志管理界面与相关接口；普通用户不可见他人操作记录。  
- **性能**：高频路径（如每次检测）可采用 **异步写入** 或 **批量写入**，避免明显拖慢主业务。

**与其他模块的关系**  
需求（1）用户管理、需求（3）知识库管理等模块中的关键操作应 **调用统一日志写入接口**，保证审计链完整；检测类业务可与「检测历史」分工：检测历史侧重**业务结果**，系统日志侧重**操作与异常审计**。系统日志管理模块流程图如图 4-11 所示。
