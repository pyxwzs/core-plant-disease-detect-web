## 一、项目概述

本项目是一个基于深度学习的**植物（茶树）病虫害智能检测 Web 系统**，采用前后端一体化的架构，利用 YOLOv12 目标检测模型对植物叶片图像进行病害识别，支持**图像上传检测、视频逐帧检测、摄像头实时检测**三种模式，并配有检测历史管理、模型数据可视化、茶树病虫害知识库等辅助功能。

**核心目标：**
- 帮助农业从业者或研究人员快速、准确地识别植物病害，辅助田间防治决策。
- 验证 YOLO 目标检测算法在植物病害识别领域的工程落地能力。

## 五、模块实现详解

### 5.1 YOLO 目标检测模型训练

**训练脚本：** `other/model_train/detect/code/train.py`

```python
from ultralytics import YOLO
model = YOLO("weights/yolo12n.pt")          # 加载预训练权重（迁移学习）
results = model.train(
    data="dataset/small_dataset/data.yaml", # 数据集配置
    epochs=50,                              # 训练轮次
    imgsz=640                               # 输入分辨率
)
```

**已训练模型（生产用）的超参数（args.yaml）：**

| 参数 | 值 | 说明 |
|---|---|---|
| model | yolo12n.pt | Nano 级 YOLO12 预训练权重 |
| data | all_dataset/data.yaml | 全量数据集 |
| epochs | 200 | 训练轮次 |
| batch | 32 | 批次大小 |
| imgsz | 640 | 输入图像尺寸 |
| optimizer | auto | 自动选择优化器 |
| lr0 | 0.01 | 初始学习率 |
| lrf | 0.01 | 最终学习率（lr0 × lrf） |
| weight_decay | 0.0005 | L2 正则化 |
| warmup_epochs | 3 | 学习率预热轮次 |
| mosaic | 1.0 | Mosaic 数据增强（启用） |
| fliplr | 0.5 | 随机水平翻转概率 |
| hsv_s/v | 0.7/0.4 | 色调/亮度数据增强 |
| patience | 100 | 早停耐心值 |
| amp | true | 混合精度训练（加速，减少显存） |

**迁移学习原理：**
YOLOv12n 在 COCO 数据集上预训练，具备通用特征提取能力。在本项目中以预训练权重为初始化点，在植物病害数据集上微调（Fine-tuning），相比随机初始化收敛更快、精度更高，尤其适合小数据集场景。

### 5.2 Flask 后端服务

**核心设计：**
- 单 `app.py` 作为路由中心，所有页面和 API 路由统一管理。
- 业务逻辑拆分到 `service/` 层，`app.py` 只做参数解析与结果返回。
- 使用 `open(..., encoding='utf-8')` 读取 HTML 模板，避免模板引擎依赖（前端自包含 Vue 3）。
- 启用 `debug=True, use_reloader=True` 支持热启动（代码修改后自动重载），`extra_files` 还监听 `.html` 文件变更。

**热启动机制：**
```python
_extra = [os.path.join(_tmpl_dir, f) for f in os.listdir(_tmpl_dir) if f.endswith('.html')]
app.run(host='0.0.0.0', port=5011, debug=True, use_reloader=True, extra_files=_extra)
```

### 5.3 图像病害检测模块

**文件：** `service/detection_service.py`

**流程（详细）：**

```
前端（detect.html）
  └─ 用户选择图片 → 转 Base64（FileReader）
  └─ POST /api/detect  {model, image: "data:image/...;base64,..."}

后端（detection_service.py）
  └─ decode_base64_image()
       ├─ 去除 data URI 前缀
       ├─ base64.b64decode()
       └─ PIL.Image.open() → 转 RGB

  └─ 写临时文件（tempfile）→ YOLO 推理（imgsz=640）

  └─ 解析 result.boxes：
       ├─ xyxy 坐标（像素级）
       ├─ conf 置信度
       ├─ cls → CLASS_NAME_MAPPING 映射中文名
       └─ 计算归一化坐标 bbox_normalized

  └─ 按置信度排序 → draw_detection_boxes()
       ├─ OpenCV 绘制矩形框（cv2.rectangle）
       ├─ 类别标签文本背景（cv2.rectangle 填充）
       └─ 结果图 → base64 返回

  └─ 返回 JSON：{detections[], total_detections, detection_image(base64)}

后续（app.py）
  └─ save_detection_record() 写入历史
```

**模型缓存机制：**
```python
_model_cache = {}  # 模块级全局字典
def load_model(model_name):
    if model_name not in _model_cache:
        _model_cache[model_name] = YOLO(model_path)  # 首次加载耗时
    return _model_cache[model_name]  # 后续直接返回缓存
```
同一进程内，模型只加载一次，避免每次请求重复磁盘 I/O 和内存分配。

### 5.4 视频病害检测模块

**文件：** `service/video_detection_service.py`

**核心设计：多线程异步处理 + SSE 流式进度推送**

```
前端（video-detect.html）
  └─ 用户选择视频 → 点击「开始视频检测」
  └─ POST /api/video/process（multipart/form-data 上传视频文件）

后端（app.py）
  └─ 保存视频到临时目录
  └─ 生成 session_id（UUID）
  └─ start_video_processing(session_id, model_name, video_path)
       └─ 创建 VideoProcessor 实例
       └─ 启动 Thread → process_video_async()

  └─ 返回 SSE 流（Response(stream_with_context(...))）
       ├─ 每 500ms 轮询 get_processing_status(session_id)
       └─ 推送 data: {...进度%...}\n\n

process_video() 流程：
  └─ cv2.VideoCapture(video_path) 读取视频元信息
  └─ 初始化 VideoWriter（优先 avc1，失败降级 XVID → mp4v）
  └─ 逐帧循环：
       ├─ detect_objects_in_frame(model, frame)
       │    └─ model(frame, verbose=False)[0].boxes
       ├─ draw_detection_boxes_on_frame(frame, detections)
       └─ out.write(annotated_frame)
  └─ 汇总统计：class_statistics、frame_details（每帧检测结果）
  └─ 保存输出视频

后续（app.py）
  └─ frame_details 完整保存到历史（save_video_history_record）
  └─ SSE 流中仅发送前 100 帧数据（避免前端 DOM 压力）
```

**为什么用 SSE 而不是 WebSocket？**
- SSE 是单向服务端推送，实现简单，Flask 原生支持，不需要额外库。
- 视频处理场景属于「服务端生产数据，客户端消费进度」的典型单向场景，SSE 足够胜任。

### 5.5 实时摄像头检测模块

**文件：** `service/realtime_detection_service.py`

**流程：**

```
前端（realtime-detect.html）
  └─ getUserMedia → <video> 实时预览
  └─ 定时器（约每 500ms）→ canvas.drawImage → toDataURL → base64
  └─ POST /api/realtime/detect {model, image}

后端（realtime_detection_service.py）
  └─ decode_base64_image()
  └─ model(frame, verbose=False) 推理（直接传 numpy 数组）
  └─ 返回当前帧检测结果

性能优化：
  └─ 前端节流（500ms 间隔），避免并发请求堆积
  └─ 仅在检测到病害时写历史（save_realtime_history_record）
  └─ RealtimeDetectionSession 维护会话统计（累计检出次数、最常见类别）
  └─ start_cleanup_task() 后台线程定期清理超时会话（防内存泄漏）
```

### 5.6 检测历史记录模块

**文件：** `service/detection_history_service.py`

**存储设计：**

```
data/detection_history/
  ├─ index.json          ← 所有记录的摘要（轻量），用于列表展示
  │   [
  │     {
  │       "id": "1776143091311_a8162808",
  │       "source": "image",          ← 类型：image/video/realtime
  │       "created_at": "2026-...",
  │       "model": "ready-model",
  │       "summary": "检出2处病害：茶炭疽病...",
  │       "total_detections": 2,
  │       "preview_filename": "xxx.jpg",
  │       "top_detections": [           ← 供卡片显示百分比条
  │         {"rank":1, "class_name_zh":"茶炭疽病", "pct_val":73.5}
  │       ]
  │     }, ...
  │   ]
  │
  ├─ records/
  │   └─ <id>.json       ← 单条完整记录（含所有检测框细节）
  │
  ├─ images/             ← 原图（xxx.jpg）+ 标注图（xxx_annotated.jpg）
  └─ videos/             ← 原始视频（xxx_orig.mp4）+ 检测后视频（xxx.mp4）
```

**ID 生成策略：**
```python
timestamp_ms = int(time.time() * 1000)   # 毫秒时间戳
md5_prefix = hashlib.md5(image_bytes).hexdigest()[:8]  # 内容 MD5 前 8 位
record_id = f"{timestamp_ms}_{md5_prefix}"  # 例：1776138564542_cc746fda
```
- 毫秒时间戳保证唯一性与可排序性。
- MD5 前缀关联原始图像内容，便于溯源。

**视频帧详情分页 API：**
大型视频可能有数千帧，直接返回会导致 JSON 过大（测试中某记录达 2 万行）。因此：
- `GET /api/detection-history/<id>` 默认不含 `frame_details`
- `GET /api/detection-history/<id>/frames?page=1&size=50` 按需分页加载

## 九、答辩常见问题及参考答案

### 【技术选型类】

**Q1：为什么选择 YOLO 而不是分类网络（如 ResNet、VGG）做病害检测？**

> 植物病害在叶片上通常以"斑块"形式局部出现，一张图像中可能同时出现多处病害区域，且位置不固定。**分类网络只能给出"图像级"标签，无法定位病害位置和数量**。YOLO 是端到端的目标检测网络，能同时完成**定位（Bounding Box）与分类（Class + Confidence）**，更符合实际应用需求。此外，YOLO 是单阶段检测器，推理速度快，适合实时检测场景。

**Q2：为什么选择 YOLOv12n（nano 版本）而不是更大的模型？**

> 本项目运行于 GTX 1650（4GB 显存）的消费级 GPU 上。YOLOv12n 是 Nano 级别，参数量最小（约 3.2M），在有限硬件条件下能保证推理速度（实时摄像头 ~500ms/帧可接受）。在数据集规模有限（2000+ 张）的情况下，过大的模型容易过拟合；较小的模型结合迁移学习，实际精度已能满足演示需求。

**Q3：为什么后端用 Flask 而不是 Django 或 FastAPI？**

> - Flask 轻量，适合快速原型和毕业设计规模；
> - 本项目接口数量适中（约 20 个路由），不需要 Django 的 ORM 和 Admin 等重型功能；
> - Flask 的 SSE（流式响应）实现比 FastAPI 更直接，适合视频处理进度推送场景；
> - 团队熟悉度高，能快速实现和调试。

**Q4：为什么前端不使用 Vue CLI / Vite 构建，而是用 CDN？**

> 项目前后端通过 Flask 直接 serve HTML，无需 Node.js 构建环境，降低部署复杂度。CDN 引入 Vue 3 + Element-Plus 对于单体应用足够，不影响功能完整性。若后续需要工程化扩展，可迁移到 Vite 构建方案。

**Q5：数据为什么不用数据库存储，而是 JSON 文件？**

> 系统用户数量极少（演示/单机环境），检测历史记录量有限（百至千条量级），JSON 文件存储成本低、无依赖、便于查看调试。若部署到多用户生产环境，需要迁移到 SQLite 或 MySQL，增加并发控制和事务支持。

### 【模型与算法类】

**Q6：YOLO 的核心原理是什么？**

> YOLO（You Only Look Once）是**单阶段目标检测算法**。将输入图像划分为 S×S 的网格，每个网格负责预测落在其中的目标的边界框（x, y, w, h）和置信度，以及目标类别概率。一次前向传播即可完成检测，相比 Faster RCNN 等两阶段方法速度更快。
>
> YOLOv12 引入了 **Attention 机制**（区域注意力模块），提升了对小目标和密集目标的检测能力。

**Q7：什么是迁移学习？你是怎么用的？**

> 迁移学习是将在大数据集（如 COCO，120 万张图，80 类）上预训练的模型权重作为初始化，在目标数据集上继续训练（Fine-tuning）。预训练权重已学到边缘、纹理、形状等通用特征，在小数据集上微调时能快速收敛。本项目加载 `yolo12n.pt`（COCO 预训练），在 2000+ 张植物病害图像上训练 200 个 epoch，最终得到 `best.pt`。

**Q8：如何评估模型性能？主要指标有哪些？**

> - **mAP（mean Average Precision）**：各类别 AP 的平均值，综合衡量精度和召回率，是目标检测最主要的评估指标。
> - **mAP@50**：IoU 阈值为 0.5 时的 mAP。
> - **mAP@50-95**：IoU 从 0.5 到 0.95 步长 0.05 的平均 mAP，更严格。
> - **Precision**：检测到的目标中真正是目标的比例。
> - **Recall**：所有真实目标中被成功检测的比例。
> - 本项目通过 `val.py` 在测试集上运行评估，结果存储在 `测试集精度.txt` 中。

**Q9：训练时用了哪些数据增强，为什么？**

> | 增强方式 | 目的 |
> |---|---|
> | Mosaic 拼接 | 将 4 张图拼成 1 张训练，大幅增加背景多样性，提升小目标检测 |
> | 随机翻转 | 增强模型对方向不变性的学习 |
> | HSV 颜色抖动 | 模拟不同光照、天气下的图像颜色变化 |
> | 随机缩放 | 增强模型对不同拍摄距离（目标大小）的鲁棒性 |
> | Random Erasing | 随机遮挡部分图像，防止模型对局部特征过拟合 |
> 这些增强都是 Ultralytics 框架在训练时自动应用的，通过 `args.yaml` 中的超参数控制。

**Q10：模型的检测置信度阈值是多少？如何处理误检和漏检？**

> 默认使用 Ultralytics 的推理阈值（conf=0.25，iou_nms=0.7）。置信度低于阈值的检测结果会被过滤。
> - **误检（False Positive）**：通过提高 conf 阈值减少；
> - **漏检（False Negative）**：通过降低 conf 阈值增加召回，但可能引入更多误检；
> - 实际应用中需要根据场景需求权衡精度和召回率。

### 【系统设计类】

**Q11：视频检测为什么用 SSE（Server-Sent Events）而不是轮询或 WebSocket？**

> - **轮询**：客户端定时请求，会产生大量无意义请求，浪费带宽和服务器资源。
> - **WebSocket**：双向通信，实现复杂，需要额外的握手协议，对于「服务端单向推送进度」场景属于过度设计。
> - **SSE**：HTTP 长连接，服务端单向推送，实现简单（Flask 的 `Response` 对象直接支持），浏览器原生支持 `EventSource`。视频处理是典型的单向数据流场景，SSE 完全满足需求。

**Q12：检测历史的 record_id 是怎么设计的，有什么考虑？**

> 格式：`{毫秒时间戳}_{图像MD5前8位}`，例如 `1776143091311_a8162808`。
> - 毫秒时间戳：保证 ID 唯一、可排序（自然时间序）；
> - MD5 前缀：与图像内容绑定，便于内容溯源和去重判断；
> - 8 位十六进制：碰撞概率极低（1/4294967296）；
> - 同时满足文件名安全性（无特殊字符）。

**Q13：系统如何防止路径穿越攻击（Path Traversal）？**

> 在 `safe_image_filename()` 和 `safe_video_filename()` 函数中，对文件名进行正则校验，只允许 `[a-zA-Z0-9_.-]` 等安全字符，拒绝包含 `/`、`\`、`..` 的文件名。在 PDF 下载接口中也有类似检查：
> ```python
> if not name or '/' in name or '\\' in name or '..' in name:
>     return jsonify({'code': 400, 'message': '无效的文件名'}), 400
> ```

**Q14：实时检测为什么不保存每一帧到历史？**

> 实时检测帧率高（约 2 帧/秒），若全部保存会在短时间内产生大量重复或近似的历史记录，占用大量磁盘空间。因此设计为**仅在检测到病害时才写入历史**（`save_realtime_history_record` 在 `total_detections > 0` 时才调用），减少噪音记录。

### 【创新点与局限性类】

**Q15：本项目的创新点是什么？**

> 1. **三种检测模式集成**：图像、视频、实时摄像头，覆盖不同场景。
> 2. **完整的检测历史管理**：带可视化百分比条的卡片式历史、视频帧级分页查看、批量删除。
> 3. **视频帧详情懒加载**：针对大型视频（数千帧）采用分页 API，避免前端渲染崩溃。
> 4. **类别标签迁移映射**：将通用植物病害数据集通过 CLASS_NAME_MAPPING 映射为茶树病害场景，实现领域适配。
> 5. **热启动开发支持**：`extra_files` 监听 HTML 模板变更，大幅提升开发效率。

**Q16：系统有哪些局限性？未来如何改进？**

> **局限性：**
> 1. 数据集中英文映射为手工对应，部分类别（如 Blueberry leaf → 茶炭疽病）属于跨物种借用，严格意义上准确性有限，未来需要采集真实茶树病害标注数据集重新训练；
> 2. 使用 JSON 文件存储，不支持并发写入（多用户同时操作可能引起数据竞争）；
> 3. 用户密码以明文存储在 JSON 中，存在安全风险；
> 4. 模型在 GTX 1650 上训练，算力有限，epoch 上限和 batch size 受约束；
> 5. 实时检测帧率约 2fps，延迟较高，暂不适合高速移动拍摄场景。
>
> **改进方向：**
> 1. 采集茶园真实病害图像，构建专属数据集，提升针对性；
> 2. 引入 SQLite/MySQL 替代 JSON 文件存储；
> 3. 密码 bcrypt 加密，增加 JWT Token 认证；
> 4. 部署到云端 GPU，提升推理速度；
> 5. 模型量化（INT8）或转换为 ONNX，提升 CPU 推理速度；
> 6. 增加检测结果的防治建议知识库，形成「检测 + 防治」闭环。

**Q17：如果让你重新设计这个系统，你会做哪些不同的选择？**

> - **数据层**：使用 SQLite 替代 JSON 文件，支持复杂查询和并发写；
> - **后端框架**：考虑使用 FastAPI，原生异步支持，自动生成 API 文档（Swagger）；
> - **前端构建**：使用 Vite + Vue 3 组件化开发，代码维护性更好；
> - **模型服务**：将 YOLO 推理封装为独立的模型服务（如 Triton Inference Server），与 Web 服务解耦，支持水平扩展；
> - **安全性**：从第一天开始就引入 JWT 认证和密码哈希，而不是在原型阶段使用明文存储。
